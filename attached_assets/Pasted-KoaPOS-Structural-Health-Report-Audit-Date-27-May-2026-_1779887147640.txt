KoaPOS — Structural Health Report
Audit Date: 27 May 2026 | Scope: Full codebase (frontend + backend)

Critical Blockers
Data lost on every page refresh — in-memory state only, no DB or localStorage

These pages use useState([]) with a module-level let nextId = 1 counter. Every record created is wiped on navigation or refresh. No backend API calls are ever made.

File	Feature	Severity
artifacts/koapos/src/pages/app/products-recalls.tsx	Product Recalls	🔴 Critical
artifacts/koapos/src/pages/app/products-pre-orders.tsx	Pre-Orders	🔴 Critical
artifacts/koapos/src/pages/app/products-return-auth.tsx	Return Authorisations	🔴 Critical
artifacts/koapos/src/pages/app/products-brands.tsx	Brands	🔴 Critical
artifacts/koapos/src/pages/app/products-tags.tsx	Tags	🔴 Critical
artifacts/koapos/src/pages/app/products-suppliers.tsx	Suppliers	🔴 Critical
Root cause: Backend routes (/brands, /tags, /suppliers) were built in artifacts/api-server/src/routes/ and registered in routes/index.ts, but were never added to lib/api-spec/openapi.yaml. Because the OpenAPI spec is the source of truth for codegen, no React Query hooks were ever generated for them, and the frontend pages were built with pure in-memory state as a stopgap. The gap was never closed.

Specific evidence:

products-brands.tsx — imports zero items from @workspace/api-client-react
High Severity — Data Persists Only Per-Browser (localStorage)
These pages store all user data in localStorage. Data is silently lost when the user clears their browser, logs in from a different device, or opens a private window. None commit to the database.

Broken integrations — UI exists, credentials missing

Integration	Backend route	Missing env vars
Xero OAuth	artifacts/api-server/src/routes/xero.ts:115	XERO_CLIENT_ID, XERO_CLIENT_SECRET
xero.ts line 168: const configured = !!(process.env.XERO_CLIENT_ID && process.env.XERO_CLIENT_SECRET) — the UI will tell users Xero is "not configured" and the Connect button will fail silently. The full Xero sync implementation (contacts, transactions, purchase orders) is complete and functional once the env vars are provisioned.

The broader Integrations page (management-integrations.tsx) catalogues OAuth providers (Google Drive, Meta, Twitter, LinkedIn, TikTok, QuickBooks, Stripe, etc.) — credentials storage in the vault table is wired, but the actual OAuth token exchange for each provider requires its own CLIENT_ID / CLIENT_SECRET pair in env, none of which are set.

Optimisation Notes — Orphaned & Structural Issues
Orphaned page file (exists on disk, never imported or routed)

artifacts/koapos/src/pages/app/management-marketing-generators.tsx (308 lines) — this file exists and contains a working QR code / shortlink settings UI, but it is never imported in App.tsx and has no registered route. It is not reachable by any user. It partially overlaps in purpose with marketing-qr-codes.tsx and marketing-shortlinks.tsx.

Duplicate referral pages

Two separate files cover customer referrals with different scopes but the same concept:

artifacts/koapos/src/pages/app/marketing-referrals.tsx (347 lines) — customer-facing referral view
artifacts/koapos/src/pages/app/management-marketing-referrals.tsx (210 lines) — management view
Both are routed and linked. This is not a bug, but a structural note — if the referral feature grows, keeping the two in sync will require discipline.

Backend routes with no OpenAPI spec entries (no generated hooks)

Three backend routes exist in routes/index.ts and their handler files are fully implemented, but are absent from lib/api-spec/openapi.yaml:

Route file	Endpoints
artifacts/api-server/src/routes/brands.ts	GET/POST /brands, PUT/DELETE /brands/{id}
artifacts/api-server/src/routes/tags.ts	GET/POST /tags, PUT/DELETE /tags/{id}
artifacts/api-server/src/routes/suppliers.ts	Full CRUD /suppliers
Because they are absent from the spec, pnpm --filter @workspace/api-spec run codegen produces no React Query hooks for them — which is the root cause of the Critical Blocker section above.

Xero redirect URI not configurable via env

artifacts/api-server/src/routes/xero.ts line 115 builds the OAuth redirect URI using a hardcoded fallback. This will cause an OAuth mismatch error in production deployments where the domain differs from the development preview URL.

Recommended Fix Priority
Add /brands, /tags, /suppliers to the OpenAPI spec, run codegen, then wire the generated hooks into the three frontend pages — this fixes 6 critical data-loss bugs in one sweep.
Migrate management-registers.tsx to a DB-backed registers table (highest-impact single localStorage migration — 21 writes, core operational data).
Migrate the Marketing module to DB (landing pages, campaigns, templates, QR library are the most business-critical).
Set XERO_CLIENT_ID / XERO_CLIENT_SECRET in the environment to activate Xero sync.
Delete or route management-marketing-generators.tsx — either remove it or add it to App.tsx at /management/marketing/generators.
Remove the three "Coming soon" staff stubs from the nav, or implement them.
